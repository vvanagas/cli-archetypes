rule:
  default: Attach by archetype unless dropped by the control's floor or excluded by the task.
  conditional: Attach only if the named trigger is present in the change.
  floor_drop: Even when attached, drop a control if its `floor` (server-controls.txt) holds.
  control_pairing:
    C04_with_C05: >
      Attach C04 (typed Result union) wherever the caller distinguishes two or
      more RECOVERABLE outcomes, paired with C05's boundary mapping. A single
      success/terminal-failure shape does not earn C04; exit-code-shaped CLIs
      (C05 + C13) and thin boundary handlers may satisfy the distinction via
      C05 alone — recorded per archetype, never assumed.
  forbidden_by_default: Do not attach. If genuinely needed, the archetype is
    probably misdiagnosed — reclassify; otherwise record an explicit justification.
  exception_policy:
    forbidden_control_used_without_record: gate_error
    accepted_forms:
      - reclassify_to_an_archetype_whose_default_or_conditional_includes_it
      - record_justification_referencing_the_specific_trigger_and_mode
  archetype_inclusion:
    meaning: >
      An archetype is expensive — it multiplies the map, the fusion
      combinatorics, and the classification step every change passes through.
      A control earns inclusion by recurrence alone; an archetype must pass
      BOTH conditions.
    conditions:
      recurrence: seen more than twice in real systems, same bar as controls
      irreducibility: >
        the candidate's default control set is NOT reproducible as conditional
        triggers on an existing archetype without distorting that archetype's
        meaning ("fails in a way nothing else does", made mechanical)
    outcomes:
      - promote_new_archetype          # both conditions pass
      - add_conditional_to_existing    # recurrent but reducible
      - reject_or_route_to_human       # unique but rare, or neither
